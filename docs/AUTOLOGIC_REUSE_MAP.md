# AutoLogic Reuse Map

## 1. Purpose

This document maps the current repository's real interfaces and artifact formats to the frozen AutoLogic design. “Modify” refers to the future implementation phase; this design-only phase changes none of the listed production files.

## 4. Required New Behavior and Why It Cannot Be Safely Patched Into LogicRAG

| Required AutoLogic behavior | Why existing interface is insufficient | Future owner | Old behavior protected by |
|---|---|---|---|
| Leaf-only executable state mapping | Current template contains root, child, and leaf nodes. | `autologic/adapters.py` | Select leaves by default; root/child nodes remain provenance and are never flattened/executed. Explicit executable title leaves remain eligible. |
| Two-report union state space | Current `build_global_template()` retains matched/common states only. | `autologic/dfa_builder.py` | Merge matched pairs, retain unmatched A leaves, retain unmatched B leaves, then apply `state_support_threshold` inclusively and project each sequence separately. |
| State `action` and source support fragments | Current normalized state drops source spans and has no separate `action`. | `autologic/adapters.py` | MVP maps `action` from `content_guideline`; excerpts come from an AutoLogic field or CSV section. Extra grounding LLM pass is optional Phase 2. |
| Separate thresholds | Legacy learner names semantic similarity `theta`, while AutoLogic also needs frequency support. | `autologic/dfa_builder.py` | AutoLogic exposes `semantic_merge_threshold`/`rho`, `state_support_threshold`, and `transition_support_threshold`/paper `theta`; old parameter stays untouched. |
| State/edge document support filtering | Current edge builder counts pairs but does not filter them by support threshold. | `autologic/dfa_builder.py` | Old `frequency/support_documents` output remains unchanged. MVP uses two-report counts; complex arbitrary-`n` filtering is Phase 2. |
| Conditional finite symbols | Current transitions have no condition or symbol. | `autologic/condition_induction.py` | New `dfa.json` schema rather than extending old templates. |
| Total DFA semantics over a sparse table | Old templates store only observed edges. | `autologic/validation.py`, `models.py` | Persist valid non-rejection edges only; all missing `(state,symbol)` pairs map implicitly to `s_bottom`, which normal runtime may not select. |
| Evidence-qualified `COMPLETE` | One observed successor does not prove unconditional completion. | `autologic/dfa_builder.py` | Use `COMPLETE` only with evidence; MVP simplifications are listed in `single_successor_unconditional_assumptions`. |
| `s_0`, sink-only `F`, and `END` | Old template has no DFA initial/final fields and historical last-state frequency can conflict with outgoing edges. | `autologic/dfa_builder.py` | Finals are reachable valid-edge sinks only. Optional termination uses `END` to a dedicated terminal control state; the branching source is not final. |
| Honest terminal-state schema | Terminal control states do not correspond to historical nodes. | `autologic/models.py`, `validation.py` | Terminal states require zero support, empty support documents/materials/action, optional empty provenance, final status, no outgoing valid edge, and no generated body; never fabricate provenance. |
| Full-DFA online traversal | Current generator loops over sorted nodes and ignores edges. | `autologic/executor.py` | Do not subclass or alter `LogicRAGReportGenerator.generate()`. |
| Per-state retrieval during traversal | Current `run_data_plugin()` resolves an entire query subtree before generation. | `autologic/adapters.py` and executor | Reuse lower-level resolver/client/fetch functions only. |
| Runtime symbol grounding | Current generator has no transition classifier or structured condition evidence. | `autologic/executor.py` | Reuse only `ChatClient` and materials formatting. |
| Conditional single-edge and `NO_MATCH` | Current generator has no condition execution semantics. | `autologic/executor.py` | Direct-select only `COMPLETE`, `END`, or `metadata.unconditional: true`. | Cycle-safe ordered outputs | Current node-output dictionary cannot represent repeated state visits. | `autologic/executor.py` | Separate filenames and ordered step schema. |
| Required CLI and keyless branch demo | Existing client has only the LogicRAG pipeline. | `autologic_client.py`, fixtures | Add `demo`, `build`, `generate`, `run`, optional `validate`; demo proves `PRICE_UP` and `PRICE_DOWN` with no keys. |
| Arbitrary `n` aggregation | Current learner is pairwise. | Phase 2 `autologic/dfa_builder.py` | Explicitly deferred; not a prerequisite for fixture execution or the Phase 1 two-report workflow. |

## 5. Compatibility Test Obligations for the Future Implementation

Before AutoLogic is considered minimally runnable, tests must establish:

1. Running the existing `logicrag_client.py --help` and stage CLIs produces the same arguments and defaults.
2. Existing `logicrag_outputs/*.json` continue to load with current modules.
3. AutoLogic imports no symbol from `query_processing.py`.
4. `RequiredMaterialResolver.build_specs()` receives only a one-current-state compatible payload during execution.
5. `IFINDDataClient` is reused and no second credential or request implementation exists.
6. `ChatClient` environment-variable precedence remains intact and no API key is written to artifacts.
7. Only leaf nodes become executable states by default; root/child nodes never appear in the execution path.
8. Duplicate `(source, symbol)` pairs are rejected before execution, and missing pairs totalize to implicit `s_bottom`.
9. Runtime chooses a valid outgoing symbol or returns `NO_MATCH`; the latter maps to `s_bottom`, records `CONDITION_NOT_SATISFIED`, and stops non-successfully.
10. Writing states require positive support, non-empty source documents/action, and non-empty provenance.
11. Terminal states require zero support, empty documents/materials/action, optional empty provenance, final status, no outgoing edge, and no generated body.
12. `COMPLETE` is evidence-supported or recorded in `single_successor_unconditional_assumptions`.
13. A sole `PRICE_UP` edge is still grounded; false evidence returns `NO_MATCH` rather than selecting the only edge.
14. Multiple outgoing conditions are mutually exclusive where practicable, cover common evidence, and return `NO_MATCH` when all are false.
15. A factual `NO_MATCH` records `CONDITION_NOT_SATISFIED`, preserves generated content, and cannot be overridden by support fallback.
16. Final states have no outgoing valid transitions; optional early termination uses `END` to a terminal control state.
17. Persisted conditions are verifiable without target identity or target semantics.
18. A branching source chooses only from its finite outgoing symbol set and records evidence/fallback.
19. For `A -> B` and `A -> C`, unmatched `B` and `C` remain when support threshold is `0.5`, producing distinct `PRICE_UP` and `PRICE_DOWN` edges.
20. Keyless `demo` executes distinct `PRICE_UP` and `PRICE_DOWN` paths without one scenario overwriting the other.
21. Demo outputs are isolated under `demo/price_up/` and `demo/price_down/`, each with all five canonical artifacts.
22. `generate` directly executes an existing `dfa.json` without rebuilding.
23. Query changes topic/date/scope but does not create a subtree or precomputed full path.
24. Final writing-state content is included before stopping; terminal control states emit none.
25. Cycles stop at configured guards and remain representable in step-sorted `generated_states.json`.
26. Dry-run invokes neither iFinD nor chat APIs and writes the five canonical artifacts.
27. Old `logicrag_outputs` and new `autologic_outputs` never overwrite one another.

## 6. Reuse Conclusion

The reusable foundation is substantial but concentrated at the boundaries:

- directly reusable: CSV row construction, per-report state extraction/normalization, embedding provider, cosine similarity, material merging, date parsing, `RequiredMaterialResolver`, `RetrievalSpec`, `IFINDDataClient`, retrieval/result formatting, `ChatClient`, materials prompt formatting, Markdown cleaning, and summary principles;
- reusable as concepts or regression references: pairwise state matching, adjacent-state projection/counting, state prompt constraints, and output assembly intent;
- intentionally not reusable in AutoLogic runtime: query embeddings, `tau`, query subtree extraction, `run_data_plugin()` over a whole subtree, sorted-node generation, and the existing LogicRAG pipeline orchestrator;
- necessarily new for Phase 1: leaf-only adapters, matched-plus-unmatched union construction, centralized `state_provenance`, honest zero-support terminal-state validation, direct action mapping, condition normalization, sparse-table/implicit-`s_bottom` DFA validation, conditional single-edge grounding, factual `NO_MATCH` termination without support fallback, sink-only finals and `END`, full-DFA executor, cycle guards, isolated keyless two-path demo outputs, required CLI commands, and step-ordered outputs;
- deferred to Phase 2: arbitrary-`n` aggregation, incremental clustering, centroid updates, complex support filtering, and optional second-pass evidence-grounding LLM calls.

This boundary gives the smallest change surface while meeting AutoLogic's method constraints and preserving the old LogicRAG implementation.
